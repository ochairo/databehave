# Stability Contract

This document defines what `@databehave/server` guarantees across versions, in
addition to the contract inherited from
[`@databehave/schema`](../../schema/docs/stability.md).

## Public surface

The **public surface** is exactly the symbols re-exported from `src/index.ts`,
plus the runtime shape of `Config` and the JSONC config schema
accepted by `loadConfig`. Anything reachable only through
deeper paths (`@databehave/server/src/...`, `_internal`, identifiers prefixed with
`_`) is **internal** and may change in any release, including `PATCH`.

The set of public symbols is locked by `test/public-surface.test.ts`.

The `admin` admin REST endpoints (`/overrides`, `/scenarios`,
`/openapi-routes`, `/ui*`) and their request / response shapes are part
of the public surface; the versioning rules below apply to them too.

## Versioning rules

Same `PATCH` / `MINOR` / `MAJOR` rules as `@databehave/schema` apply, plus:

| Bump      | Additional triggers for `@databehave/server`                                                                                                         |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **PATCH** | Wire-level behaviour of `fetch(Request) → Response` for handlers that previously returned identical bodies. Header casing fixes.                  |
| **MINOR** | New CORS options, new mock-mode hooks, new JSONC fields with backwards-compatible defaults. Additive `admin` admin REST endpoints, additive fields on admin request / response shapes, new `x-mock-*` headers. |
| **MAJOR** | Default CORS policy changes that may close previously-permitted requests. Removal or renaming of JSONC fields. Changes to OAS path dispatch order. Removal / renaming of admin REST endpoints, breaking admin request / response shape changes, removal / renaming of `x-mock-*` headers, changes to sticky-override resolution priority. |

## Determinism contract

For a fixed config (`Config`, JSONC, OAS document) and a fixed
request, the **response body** generated by @databehave/server is deterministic
across `PATCH` and `MINOR` releases. The same per-request seeding rules
documented in `docs/seeding.md` apply.

## Mock-mode dispatch order

Within a single config, mock-mode resolves the response status and body in
this order, in this priority (first match wins):

1. Explicit `pathOverrides` entry keyed by `'<METHOD> /path'` (concrete path).
2. Explicit `pathOverrides` entry keyed by `'/path'` (any method).
3. Per-endpoint `status` set in JSONC `endpoints`.
4. `mockMode.defaultStatus` (when set).

Body resolution, once a status is chosen, follows this priority:

1. Hand-written handler body (when registered for the route).
2. OAS `responses[N]` schema match by **specificity**:
   1. Static path patterns (`/items/me`) before parametric (`/items/:id`).
   2. Within the same kind, longer paths before shorter wildcards.
   3. Lexical fallback for ties (deterministic, declaration-order
      independent).
3. Generic envelope `{ error: true, status: N }`.

This order is part of the public contract; changes are `MAJOR`.

## CORS defaults

- `cors: undefined`            → no CORS headers added (handlers fully control).
- `cors: {}`                   → permissive default for **mock development**:
  `Access-Control-Allow-Origin` mirrors the request `Origin` (or `*` if absent).
  This is intentionally lax because @databehave/server is a developer mock server.
- `cors: { origin: ... }`      → strict; the resolver function is authoritative.
- `cors: { allowHeaders: [] }` → hard lockdown; preflight omits the header.

When a deployment must run on a network where third parties can reach the
mock server, set `cors: { origin: '...' }` explicitly. The README documents
this.

## Fail-loud policy

Configuration that is structurally valid but semantically nonsensical throws
at config-load time, not at request time. Examples already enforced:

- `pathOverrides` keys that do not start with `/` or `<METHOD> /`.
- `pathOverrides` values that are not numbers.
- OpenAPI documents that fail to parse as JSON (convert YAML first via `yq -o=json`).

## Logger injection

Every diagnostic the kit emits goes through `LoadConfigOptions.logger.warn`
(or the equivalent `MockModeConfig.logger`). The kit never calls
`console.warn` / `console.error` directly. Callers may pipe to their preferred
sink (pino, winston, structured JSON, /dev/null).

The opt-in `log` feature (top-level `log` config, see
[config.md](./config.md#log)) is **separate** from diagnostics. It writes
one line per request to **stdout** by design, is off by default
(`log: false`), and is intentionally not routed through the logger
injection hook — access logs and diagnostic warnings have different
audiences and different destinations. This split is intentional and is
not a contract violation.

## Admin mode contract

When `admin` is opt-in, the following defaults are part of the
public contract — changing any default is **MAJOR**:

- `enabled: false`
- `path: '/databehave'`
- `bind: 'loopback-only'`
- `allowDestroy: true`
- `cors: 'auto'`

Additional rules:

- **`x-mock-*` header semantics.** Adding a new header is **MINOR**.
  Removing or renaming a header, or changing the ErrorMode kind a
  header resolves to, is **MAJOR**.
- **Sticky-override resolution priority.** The order
  `header > exact > path > global > existing mockMode > passthrough`
  is part of the contract — changes are **MAJOR**.
- **Scenarios file format.** Adding new top-level fields to a saved
  scenario JSON is **MINOR**. Changing the directory layout, the file
  naming, or the meaning of an existing field is **MAJOR**.
