# Auto-schema mode

> "I have an OpenAPI document. I want realistic, deterministic
> mock data. I don't want to write any TS schemas or handlers."

Set the top-level `schema:` field in `databehave.jsonc` and every
endpoint absent from `endpoints` is served from data generated by
`@databehave/schema`'s `mock()` engine — derived directly from your
OpenAPI document.

**Minimal:**

```jsonc
{
  "openapi": "./openapi.json",
  "schema": { "enabled": true }, // "enabled" gates auto-mode; defaults equivalent to { "seed": "stable" }
  "endpoints": {}
}
```

**With knobs:**

```jsonc
{
  "openapi": "./openapi.json",
  "schema": {
    "enabled": true,
    "seed": 42, // number = fixed; "stable" = derive from endpoint|query|params; "random" = fresh per request; default "stable"
    "locale": "ja",
    "arrayCount": 20 // default array length when the OAS lacks minItems
  }
}
```

## Architecture

The server walks your OAS, translates each response schema to a
`@databehave/schema` IR via `translateOasToIR` (in-server, ~150 LoC,
zero-dep), then calls `mock(ir, { seed })` per request. The
`@databehave/schema` package is loaded **only** via
`await import('@databehave/schema')` when `schema:` is set — the
server itself keeps `dependencies: {}` and `peerDependencies: {}`.

The IR is **pre-translated** at server-creation time. Every
response schema in the OAS doc is walked once during boot; per-
request work is just `mock(ir, { seed })`. This catches translator
errors loudly at startup (`@databehave/server: translateOasToIR
failed for GET /widgets responses[200]: <reason>`) instead of
serving 500s on the first hit.

Translator scope:

- Only the `200` (or smallest declared `2xx`) JSON response is
  translated for each endpoint. Other statuses fall back to the
  zero-dep generator and the per-status body table from
  [fallback.md](./fallback.md#per-status-body-generators).
- `204` responses are skipped entirely — no body, no IR.

## Stable seed derivation

`seed: "stable"` (the default) computes a per-request 32-bit FNV-1a
hash so identical requests produce byte-identical responses across
restarts and across machines:

```text
input  = `${METHOD} ${endpointPath}|${sortedQuery}|${sortedPathParams}`
seed   = fnv1a32(input)
```

- `endpointPath` is the OAS pattern (e.g. `/users/:id`), not the
  resolved live URL — `/users/1` and `/users/2` therefore generate
  different bodies, while `/users/1` always generates the same one.
- `sortedQuery` is the query string with keys sorted lexically, so
  `?a=1&b=2` and `?b=2&a=1` collide on purpose.
- `sortedPathParams` is the matched path-param map serialised the
  same way.

Snapshot tests built on top of auto-schema mode rely on this exact
input — change the URL or the param shape and the body changes.

`seed: <number>` skips the derivation and feeds the literal value to
`mock()`; every request to the same endpoint returns the same body.
`seed: "random"` calls `Math.random()` per request so bodies vary —
useful for load tests, useless for snapshots.

## Locale (forwarded, currently no-op)

The `locale` field is parsed and forwarded into the per-request call
to `mock()`, but `@databehave/schema`'s zero-dep mock engine does
not currently consume it — output is locale-agnostic regardless of
the value. The field is kept so a future `@databehave/schema`
release can pick it up without changing this server's API surface.
For locale-tied corpora today, generate fixtures up-front via
`@databehave/schema` and serve them from `endpoints` with
`response.json`.

## Install path

Auto-schema mode requires the optional peer:

```sh
npm i @databehave/server @databehave/schema
# or: pnpm add @databehave/server @databehave/schema
# or: yarn add @databehave/server @databehave/schema
```

## Missing-install error

If `schema:` is set but `@databehave/schema` is not installed,
server-creation fails fast at startup with a friendly multi-line
install hint. The native `Cannot find module` error from the
underlying `await import(...)` is suppressed (no `cause` chain) and
replaced verbatim with:

```text
[@databehave/server] Auto-schema mode is enabled in databehave.jsonc
("schema": { "enabled": true } is set), but the data-generation engine is not installed.

  npm i @databehave/schema
  # or: pnpm add @databehave/schema
  # or: yarn add @databehave/schema

This enables realistic, seeded mock data derived from your OpenAPI
document. See: https://github.com/ochairo/databehave/blob/main/packages/server/docs/openapi/auto-schema.md#missing-install-error

To keep the default zero-dep placeholder mode instead, set
"schema": { "enabled": false } or remove the "schema" field from databehave.jsonc.
```

## Precedence

Hand-written `endpoints` always override auto-mode for the same
route — auto-mode only fills paths absent from your `endpoints` map.

## Default behaviour

Omit the `schema:` field and the server keeps its existing zero-dep
placeholder generator (deterministic structural values like
`"string"`, `0`, `false`). Activating auto-mode is opt-in.
